# CLAUDE.md

本檔案為 Claude Code（claude.ai/code）在此倉庫工作時的指引。

## 專案簡介

`NotebookLM_PDF_2_PPT` — 將 NotebookLM 簡報 PDF 檔轉換成可編輯 PowerPoint 的 CLI 工具，每頁投影片以全解析度頁面渲染圖為背景，疊上實心填色矩形蓋住點陣文字，並承載可編輯、樣式比對過的文字（字級、粗體、文字色、蓋板色皆從影像估計）。

## 常用指令

```powershell
pip install -r requirements.txt        # rapidocr 首次執行會下載 ONNX 模型（需要一次網路）
python pdf2ppt.py "input.pdf"          # 完整轉換 -> input.pptx
python pdf2ppt.py in.pdf -o out.pptx --pages 1-2 --debug   # 快速迭代：指定頁 + 疊框 PNG/JSON
python tools/compare_pptx.py generated.pptx reference.pptx # 與參考簡報比對文字召回率
```

本專案沒有自動化測試；驗證方式是轉換範例 PDF 並與參考 PPTX 比對（見下方「驗證」）。

## 架構

每頁的處理管線（由 `pdf2ppt/cli.py` 調度）：
渲染（`render.py`，PyMuPDF @200dpi，同一張渲染圖同時供 OCR 與投影片背景使用）→ OCR（`ocr.py`，RapidOCR PP-OCRv5 + server 辨識模型 — 繁體中文準確度遠勝預設的 v4 mobile；`--fast` 可切回）→ 樣式估計（`style.py`）→ 選擇性行合併（`blocks.py`，預設與 DeckEdit 相同、每個 OCR 行一個形狀）→ PPTX 輸出（`builder.py`）。

以下是經過校準才得到的關鍵不變量 — 不要隨意更動：

- **兩套座標系**：PDF 頁面可以是任何尺寸（範例：1376×768 pt = 19.1 吋寬），但投影片固定為 12192000 EMU（960 pt）寬。字級必須換算到「投影片」點數空間：`px_to_slide_pt = 960 / image_width`。
- **字級來自字墨而非 OCR 框**：RapidOCR 偵測框的留白不一致。`style.py` 量測框內緊貼字形墨水的高度，再除以 `CJK_INK_RATIO = 0.91`（用 PowerPoint COM 實際渲染輸出、回測墨水高度所得的實證值）。字墨列要**分群取最重的群**（空隙 > 8px 即切群；行內真實空隙如 i 的點 ≤ ~4px）— 跨進框內的外來墨水會灌高字墨帶、放大字級、讓蓋板塗掉鄰近內容：膠囊框線的圓角/反鋸齒列佔寬 < 0.85 混進字墨（images/ 案例：離主帶 16 空白列）；上一行的底線 `_` 與下一行的字頂也會壓進框內（graph.sh 案例：各離主帶 9 空白列，門檻曾是 0.12×框高=11.4 而漏切）。
- **混合行（中英夾雜）字級取逐字 CJK 共識**（`style.py` 的 `_cjk_band_height`）：拉丁降部（g/p/y、括號、/）會伸到表意字帶底下約 0.15 em，整行字墨高除以 `CJK_INK_RATIO` 會灌高字級（第 4 頁「graph.sh 使用」：66px vs 純 CJK 同級行 58px → 18pt 應為 14pt）。每個 CJK 字各開欄窗測列範圍，**過半字落在中位數 ±4% 內才採叢集最大值，否則退回全字最大值（±2% 曾因正常字形差異讓叢集空集、退回 max 撿到被上行降部污染的字：p14「轉化為 AI 的 Guardrails」[80,75,93,75] 量成 24pt；±4% 仍排除 5% 的模糊突起）**——叢集規則同時剔除「鄰行墨水灌進單一字窗」（出 +14%）與「模糊突起」（序/譜 61px vs 四個同伴 58px，恰好跨 14/16pt 折點）；退回規則保住共識不足的雜訊行（12pt 的周郁凯字高散佈 39–47px，硬套中位數會錯殺正確的 47）。欄窗由逐字 advance 寬度按比例映射到框寬 — CTC word box 太抖不能用（「使用」的字框偏移半個字、量出 33px）；每字窗內縮 0.2 em；低墨全形標點（（）：。等）整個欄窗排除（全形（同樣降到帶底下）；窗內列範圍要套同樣的「空隙 >8 切群取最重」（graph.sh 框頂 3 列雜墨曾讓它反彈回 18pt）；窄欄窗**不可**套 `MAX_INK_ROW_FRAC`（CJK 橫筆畫會貫穿整窗被誤刪成 8pt）；結果以 `min(完整字墨高, 共識高)` 為準。**門檻：含拉丁降部字元的行（可單字錨定），或含拉丁字母且 ≥2 個表意字票的行**（p14 案例無降部字元但被上行 p/q 字莖橋接灌高；≥2 票防「2025 年」單字塌縮）。再放寬到所有 CJK 行已試過並回退 — ±3px 模糊雜訊讓其他臨界行掉一階、把原本一致的區塊拆散（p4 右註解 16→16/14/14），單 CJK 字的行（「2025 年」）更會塌到單字高度。「稀疏邊緣列裁切」（< 0.25×中位數）也試過並放棄（曾把 44pt 標題裁成 40pt）。蓋板與文字定位仍用完整字墨範圍 — 降部點陣必須被蓋住。
- **同段折行字級調和**（`blocks.py` 的 `harmonize_font_sizes`，在 `cli.py` 浮水印過濾後呼叫）：±3px 模糊雜訊跨在 12/14/16pt 折點上，同一段折行的兩行可能折到不同字級（p10「CB 詢圈案例/」12.6pt vs 折行同伴「法規或生辰八字」14.1pt，字形實際同高）。垂直緊鄰（間距 −0.6h~0.45h）、x 重疊 ≥40%、粗體/底色/文字色相同、**snap 前估值差 ≤14%** 的行歸入同組（12% 曾差 0.6% 漏掉 p11 的折行對 11.8 vs 13.5；真實 12-vs-14 對差約 19%；底色差門檻是 **25 不是 16** — 照片漸層讓折行同伴的 bg 估值漂移，p7 成果/零/研究 三行量出 180/160/153、曾被 16 拆組折成 18/20/20）；組內字級若散落在**相鄰** snap 階上就統一（多數決，平手取組內估值中位數重 snap），但**統一目標不可超過組內任一成員的 `max_fit_pt`（投影片邊緣硬天花板）** — p15 折行對曾被平手中位數重 snap 推回 32、再度衝出投影片。判斷用 `Style.est_pt`（snap 前、寬度夾制後的估值），不能用 snap 後值 — 兩行落在折點兩側時 snap 後差一整階。這個 pass 修掉的不一致橫跨 7 頁（觸發 Git Commit、N 個/Subagent、橘色[對比] 與另三色籌片、p15 終端機框四行等），全部經視覺驗證。
- **單行形狀關閉自動換行**（`builder.py`）：字級估計只要略大一點就會把整行折行、毀掉版面。
- **CJK 字型需要直接寫 XML**：python-pptx 的 `font.name` 只設定 `<a:latin>`；必須用 `_set_east_asian_font()` 注入 `<a:ea>`，否則中文會悄悄回落到佈景主題字型。
- **緞帶/色塊偵測**（`style.py` 的 fallback 分支）：當 OCR 框外圍環帶不是單一平面色時，取框內中段水平帶做雙色聚類並對各遮罩做侵蝕 — 細筆畫會消失、實心緞帶會存活；存活者為蓋板色，另一個為文字色。單純的主色佔比判斷在這裡會失敗（量化會把緞帶色拆散到多個 bin）。
- **粗體判定：≥24pt 一律粗體；≥16pt 用「模板比對」；更小用筆畫門檻**（`style.py`）：
  - **原始筆畫門檻（stroke_rel ≥ 0.13）在 16pt 以上完全重疊**：p3「Layer 3」標題（粗）量 0.108、p9「多模態輸入」（細）量 0.098 — 相對筆畫寬隨字級變大而下降、隨文字內容（拉丁 vs 密集 CJK）與對比度漂移，固定門檻救不了（使用者回報 2026-06-12：p2 核心洞見、p3 Layer 標題、p10 AI 逆向掃描全漏粗）。
  - **模板比對（`_template_bold_r`）**：該行文字用 YaHei Regular/Bold 各渲染一次（msyh.ttc/msyhbd.ttc）、縮放到同字墨高、套 σ=1.2 高斯模糊模擬 72dpi 升頻、以**該行自己的對比度相對門檻**（INK_DIST/文字-底色對比）切 ink mask、用同一個侵蝕筆畫估計器量測，r =（實測−Regular)/(Bold−Regular)。**對比相對切割正是「白字深底量得偏細」的根源**：白字深底對比（~160）比黑字白底（~235）低、固定 INK_DIST 相對切得更深、筆畫膨脹更少 — 模擬進模板後兩邊抵消（p12「1.先寫規則」深字與 p13「4. 批次整理」白字是同系列卡片標題，rel 差 35%，r 才同側）。全文校準（2026-06-12）：≥16pt 細體上限 r=0.09、粗體下限 0.17，門檻 `BOLD_R_THRESH=0.13`。σ 平移整個 r 刻度（1.0→+0.07、1.5→−0.15）但保持排序 — 門檻與 σ=1.2 綁定、不可單獨動其中一個。
  - **適用條件**：字級 ≥16pt、bg 已知、對比 ≥75、非弧形，文字先濾掉 YaHei 可能缺字形的字元（→、Ⅹ 等 .notdef 是實心塊）後 ≥2 字；w_reg > 2.5×實測 → 模板崩壞、放棄（全大寫拉丁＋模糊會黏成塊：PITCH DECK r=−2.4）。**14pt 以下模糊噪聲支配 r**（p12 同款內文散佈 −0.36…+0.10），維持筆畫 0.13 規則；14–15pt 仍算 r 但只當撤粗保護的第二意見。
- **粗體隊列投票**（`blocks.py` 的 `harmonize_bold`，在字級調和**之後**跑）：判別器在門檻附近擲硬幣，同頁同字級、由筆畫/模板判別的行組成隊列投票：**補粗體需 ≥2/3 多數**，**撤粗體需 ≤1/5**（孤立誤判如 p7 Before；曾設 ≤1/3 誤撤 Input:/OutputA 真框標題）。可翻動帶按主判指標分流（`_bold_promote_ok`/`_bold_demote_ok`）：模板判的行補粗需 r ≥ −0.05（「Karpathy Wiki 模式」r=−0.008 靠七個粗體鄰居救回 — 拉丁字模板基準偏高害它落線下）、撤粗限 r < `TPL_MARGINAL_R`(0.22)（「人類輸入：」r=0.25 是孤身在 8 行內文隊列中的真強調、不可撤）；筆畫判的行維持 [0.115, 0.15] 帶，但 r ≥ 0.28 否決撤粗（p12 14pt 卡片標題「1.先寫規則」rel 0.144/r 0.43、「3.人類給骨架」rel 0.133/r 0.31 淹在 17 行內文隊列裡，舊規則誤撤）。隊列鍵**不含文字色**（p5 四個同款標題曾各一色）。順序必須字級先調和（SKILL.md 案例）。
- **邊緣模板粗體不可鎖死折行組**（`blocks.py` 的 `harmonize_font_sizes`）：14pt 折行尾「為體系化的高價值資產。」出生 snap 成 16pt、r=0.146 恰過門檻判粗，「粗體不同不入組」曾把它擋在 14pt 細體折行同伴組外、字級卡死 16pt。r < `TPL_MARGINAL_R` 的模板粗體不阻擋歸組；組內粗體屬**嚴格少數**時邊緣粗體跟隨多數撤掉（1v1 平手不動：p4「人類專屬：」r=0.219 保住粗體開頭、其折行尾維持細體）。
- **文字垂直定位**（`builder.py`）：形狀同時身兼蓋板（必須涵蓋整個 OCR 框）與文字容器（字形必須落在點陣字墨上）。小字時偵測框上緣留白會大於行高補償、兩者衝突 — 解法是用 `margin_top` 內距解耦，絕不要移動形狀頂端。
- **同列重疊框要裁切**（`builder.py` 的 `_trim_row_overlaps`）：偵測器會把框外擴壓進同列鄰行的字形（例：「…書）」框尾蓋進「再批次…」框頭 43px），後畫的蓋板就會塗掉先畫的鄰行文字。在渲染圖上找兩段字墨間最寬的空白欄縫、把兩框各裁到縫中線兩側（各退 `COVER_PAD_PX`）；縫被實心墨水填滿時退而取重疊區中點。
- **空格還原**（`ocr.py`）：辨識模型會丟空格。拉丁字詞間的真實空格從逐字 word box 復原（`return_word_box=True`，間隙 > 行內字寬中位數的 1/3）— 但 CTC word box 邊緣不準（KEEP 中間出現過 17px 的假縫隙、句點框寬達 30px），所以候選間隙還必須通過物理檢查：間隙條中段的墨水比例 < 0.15（實測：真空格 0.00–0.10、假縫隙 0.24）**且完全空白欄帶寬 ≥ max(4px, 0.2×字寬中位數)**（實測：Git Commit 真空格 23px、p6 Transformer 的 r→m 假縫隙 0px — r 字形右下天生稀疏、墨水比例擋不住，但模糊讓字墨連續、空白帶寬抓得住）。中文字 ↔ 英數字交界一律補空格（盤古之白），因為該處的量測間隙不可靠。另外重組時必須**走訪原始文字**以保留模型自己輸出的空格 — 間隙分析只能「增加」空格、不能弄丟。
- **行尾標點復原**（`ocr.py`）：偵測器常把行尾的 。/） 裁掉。當框右側有墨水時，將行條右延 1.2×行高後以 **rec-only** 重新辨識（`use_det=False` — 小裁切跑偵測會把行打碎），只接受「補回 1–2 個 `TRAIL_PUNCT` 標點」的結果。**兩道防幻覺守衛**（p12 卡片的紅棕虛線邊框被讀成 ：、整頁長出「單次寫：」「免疫防護。：」）：(a) 行尾已是 。！？ 的行不得再補 ：；，、．；(b) 補回標點的墨水欄位若在行帶上下的行距區（±0.08h~0.5h）都有墨、而**右移一個標點寬的對照窗乾淨** → 是垂直邊框、拒收 — 對照窗是關鍵：p2 橫幅色帶的上下緣會讓行距區全寬有墨，沒有對照窗時合法的 …的書。 也被誤殺。**行尾出現開引號（「『（等）視為幻覺**（框外被切半的 。 左弧會被認成 「，p6 案例）：改重讀「行尾 5 字高寬度」的小段、以 4 字後綴比對 — 整行重讀在 3400px 長行上會丟字（翻 N 本 → 翻本）害 startswith 比對失敗；真正的折行開引號重讀仍是 「、不在 TRAIL_PUNCT 內、自動保留原文。注意：RapidOCR 的 `use_det/use_cls/use_rec` 會**跨呼叫殘留** — 每次引擎呼叫都必須三個旗標全部顯式傳入，否則 rec-only 呼叫會污染下一次完整呼叫。
- **蓋板/文字色的分層情境**（`style.py`）：只靠環帶取樣會以兩種方式失敗。(a) 文字位於灰帶之間的近白色橫條 → 環帶取到錯的帶；以「光暈修正」解決（蓋板色改取距筆畫 4–6px 的帶 — 緊貼筆畫的 3px 是被字色染色的反鋸齒區，曾經悄悄偏移過整份文件的蓋板色；且光暈色只在與環帶色相差 ≥20 時才覆蓋，同一表面上環帶色更純）。(b) OCR 框溢出實心膠囊、壓到頁面背景 → 環帶看到外圍色、整顆膠囊被當成墨水；以 `ink.mean() ≥ 0.45 且 survival ≥ 0.45` 偵測，再用 k=3 色彩聚類重推底/字色（最實心 = 膠囊、最細碎 = 筆畫），且**重推底色必須與環帶色差 ≥40 才採用** — 密集粗體大字（p15「把精力保留…」ink.mean 0.46）會誤闖此分支，把反鋸齒灰殼當文字色、整行刷成灰。文字色一律只平均「筆畫核心」（離底色最遠的前 30% 墨水像素），因為反鋸齒會把筆畫邊緣拖向背景色。
- **弧形文字**（第 3 頁緞帶橫幅）：弧度只能從**救援片段中心**擬合（`_arc_from_fragments`），純影像逐欄追蹤法已嘗試並放棄 — 平行的兩條弧線框深度交疊、互相污染、連符號都會判反。高分的鄰行用「夥伴傳播」繼承曲率（同半徑 → 弧深隨寬度平方縮放）。輸出端 PowerPoint 的 `prstTxWarp`（textArchUp/Down）也已嘗試並放棄：其下沉量 ≈ 0.13×框高（實測校準），但字級/路徑縮放行為不透明，長文字會繞圈重疊 — 改用 **3–6 段切線文字方塊**沿拋物線排列（局部切線角 = atan(2·弧深·t/半寬)），色塊用 12 條沿弧條帶（縱向邊距 0.3 字高吸收拋物線 vs 圓弧的側翼偏差）。段落切割用 em 寬度加權並貼齊空格。拋物線錨點**不能用 bbox**（救援片段會把框灌高：中段錨點實測低了 39px、弧深 172px vs 實際 ~97px）— `_arc_geometry` 用渲染圖在中心與 ±0.65 翼側量「文字色字墨 run」重新錨定，每個探測點取離預測最近的 run（同一緞帶有兩行弧線、視窗會同時看到兩條字墨帶）。條帶高 1.5×字高（2.2× 會把上緣的柔和漸層蓋成一條硬邊 — 使用者回報過）；垂直範圍再以 `_ribbon_limits` 夾進緞帶本體（只縮不放）：字墨中斷的橋接只允許在距切線中心 ±0.9 字高內，區外只容 10px — 否則會跨過「邊緣帶+白縫」橋接到同色的虛線、夾製整個失效；並沿條帶取 3 個樣本欄、在切線座標系取極值，旋轉後的角落才不會凸出緞帶（曾在翼側產生凸塊）。相鄰條帶的角差會在底部張開楔形縫，條帶寬度要加 `strip_h × Δslope` 的重疊量補縫。
- **傾斜文字，兩種情況**：
  1. 偵測器回傳「旋轉 quad」（例如第 5 頁 26° 的金字塔色帶）：`ocr.py` 把角度/中心/轉正尺寸存進 `Line`；`style.py` 用 PIL `Image.QUAD` 轉正後的貼片取樣顏色與墨水（若改用軸對齊外接框取樣會混入周圍顏色 — 那個 bug 曾產生蒼白的蓋板）；`builder.py` 輸出旋轉形狀（`shape.rotation`，兩者皆順時針為正，PowerPoint 以形狀中心旋轉）。
  2. 偵測器把傾斜文字合併成「軸對齊」框（第 3 頁的弧形緞帶）：該行分數會 < 0.75 並觸發旋轉救援 — 將外擴裁切以 ±3/6/9° 重新 OCR，只保留「自身高度帶跨過裁切中心」的片段（否則鄰行會混入），在「長度不超過原文 2 倍」的前提下取最長、分數最高的組合。PCA 與投影剖面去斜都在這種雜訊裁切上失敗過；暴力法勝出。當 quad 已帶角度（情況 1）時跳過救援。救援成功時同時重建旋轉矩形（用 `p_in = c_in + R(+θ)(p_out − c_out)` 映射回原圖，已數值驗證），輸出形狀同樣旋轉 — 高度要用片段高度的**中位數**，不能用 quad 聯集（弧形片段會把聯集灌爆：243px vs 實際 ~80px）。
- **簡體混入修正**（`ocr.py`）：辨識模型偶爾吐出簡體字形（惡→恶）。OpenCC **只套用在繁簡混雜的行** — 純簡體的行是刻意呈現的內容（例如第 4 頁畫面中的搜尋詞「掌控习惯」），必須保留。行級平手（規则）純文字無法判斷，預設轉繁體。字典必須用 **s2tw 而非 s2t**（使用者回報 2026-06-12）：s2t 是正體舊形字典，會把台灣慣用的合法繁體字也改寫 — 「越吃重」被轉成「越喫重」；s2tw 保住 吃、照樣修 恶→惡。**整行 CJK 全滑成簡體時混雜測試不會觸發**（p3「AI 执行：」、p9「LLM 合规」— 拉丁字母不影響 t2s 往返、行被當成純簡體保護）：`_confirm_simplified_strays` 用頁面詞彙佐證解套 — 轉繁後「有變動的 CJK 字串」（2–4 字）逐一在同頁高分行詞彙中找到才採用（執行 由同頁「…AI 執行的自動化編譯…」佐證）；找不到佐證就維持原樣，刻意呈現的純簡體內容（掌控习惯）因此仍受保護。
- **多色行**（`style.py` 的 `_split_color_runs` + `builder.py` 的 `_split_text_runs`）：用逐字 word box 取每個字的顏色並分組成 run（合併門檻 ≤45、只有色差 ≥60 才真正分段 — 不對稱門檻防止模糊雜訊拆散單色行）。run 的字數涵蓋「去空格」後的文字；`Line.char_boxes` 攜帶字框，凡是破壞字元 1:1 對應的轉換（救援）都要把它設為 None。**夾在兩段同色 run 之間的單字 run 是 CTC 字框抖動**（p4 Idempotent 的尾 t 字框壓到「(」字縫、取到反鋸齒灰 150）— 併回鄰段，不要讓單字雜訊拆 run。
- **無蓋板救援**（`style.py`）：斜置籌片（p8 等距視角的綠色平行四邊形）的軸對齊框同時罩到籌片+頁底+鄰籌片，環帶混色、雙色聚類也失效（兩個叢集都是實心面、無侵蝕存活差）→ bg 判 None → 透明文字疊在點陣數字上出現重影。救援：取字墨帶光暈（dilate 3→6 的環）主色，佔比 ≥0.65 即當蓋板色；漸層/插畫區主色不純、維持無蓋板。
- **頁面詞彙校正**（`ocr.py` 的 `_vocab_correct`）：最小的籌片字退化到模型極限（p9 資訊 讀成 黃訊 0.66、驅動 讀成 驅勤），但同頁通常有大字高分行寫著同樣的詞（商業驅動>資訊架構>系統建設 0.85）。低分（<0.8）2–4 字純 CJK 行若與頁面詞彙（高分行的 2–4 字子串）恰差一字就校正。**候選以頻次裁決、嚴格最高頻才採用** — 跨詞邊界子串會污染候選（架的 來自 框架的…），但真詞會在頁面重複出現（架構 在 資訊架構+RAG架構）。
- **同籌片漏行救援**（`ocr.py` 的 `_rescue_sibling_bands`）：偵測器整行漏掉小籌片的第二行（p9 資訊 下面的 架構，det 任何解析度/模型都看不見 — 試過 max_side_len 4096 與 det server 模型，更糟）。對 2–4 字小 CJK 行上下探帶、rec-only 重讀，**只在頁面詞彙能驗證（精確或差一字）時採用** — 未驗證就插入會蓋掉正確點陣字卻顯示錯字，比漏掉更糟（架構 band 讀出來是 架樽/架模）。
- **警告圖示修正**（`ocr.py`）：行首的 ⚠ 三角形會被辨識成 'A'。判別器：圖示底部是一條連續墨水帶（佔框寬 0.93）、真正的 A 是兩隻分開的腳（0.24）；門檻 0.6，只套用在「行首 A、後接空格或中文」。
- **實心圖示誤認單字母**（`ocr.py` 的 `_looks_like_filled_icon`）：彩色圖示會被認成字母（p15 綠色循環箭頭 → 分數 0.86 的「S」）。判別：字形墨水只有一種顏色，圖示是「填充+描邊」兩個強色叢集（各 ≥25%、距離 ≥60）。兩個陷阱：背景色**不能用環帶**（緊貼圖示的環帶會落在圖示上、把背景估成圖示色），要用框內主色；反鋸齒也會形成第二叢集，但 AA 色落在「背景↔字色」連線上（殘差 < 45），圖示填充色偏離該線 — 必須做共線檢查（TSV 曾被誤判）。只套用在「單一拉丁字母」的偵測行。
- **寬度量測按字種分字型**（`style.py` 的 `_measure_em`）：混排行的拉丁段用 Arial、CJK 段用 YaHei 分段加總 — 全用 YaHei 量會高估輸出寬度（YaHei 拉丁比 Arial 寬 ~9%），夾制把 p5「Git Hook 自動化」從真實 24pt 掐到 20。單字呼叫與 2 字短串維持原規則（見上）。注意**來源字型的拉丁還要再窄 ~14%**，Arial 也吃不下時靠懸崖防護（見下）。

## 驗證

1. `python pdf2ppt.py "SOURCE.pdf" -o generated.pptx`
2. `python tools/compare_pptx.py generated.pptx "SOURCE.pptx"` — 真實內容行預期 100% 召回；被 `drop_illegible_lines` 刻意丟棄的插圖雜訊行（參考檔若未過濾會含有）列為 MISS 屬預期，逐條確認 MISS 都是雜訊/留圖行即可
3. 視覺驗證：用 PowerPoint COM 把投影片匯出成 PNG（`$pp = New-Object -ComObject PowerPoint.Application; ...Slides.Item($i).Export(path,"PNG",1376,768)`），與 `pymupdf` 的頁面渲染圖上下並排比對，比對圖放 `verify/`。
4. **不要重新產生 `SOURCE.pptx`**（使用者指示 2026-06-12）：每輪修正只產生並保留 `generated.pptx` 給使用者自行驗證；`SOURCE.pptx` 由使用者管理、僅作為步驟 2 的比對基準。

- OCR 推論裝置：`--device` 預設 auto（DirectML > CUDA > CPU 依可用性）。本機裝的是 `onnxruntime-directml`（取代 CPU 版 onnxruntime）。實測 Intel Arc 140V：**DirectML 全份 50s vs CPU 210s（快 4.2×）**。注意**首次** DML 執行要付一次性的驅動 shader 編譯（該次 ~213s、看似與 CPU 同速）——編譯結果由 Intel 驅動存碟快取，之後每次執行都命中快取；不要被首跑數據騙了下「GPU 沒用」的結論（我們差點下錯）。rec_batch_num 加大反而變慢（16: 0.98s vs 6: 0.80s/頁），維持預設
- Windows 11、Python 3.14；主控台是 cp950 — 印中文的腳本需要 `python -X utf8`（cli.py 自己會重設 stdout）
- PowerShell 會吃掉 `python -c` 單行指令中的雙引號；這類情況請改用 Bash 工具或腳本檔。git commit 訊息中的雙引號也會被截斷 — 訊息避免使用雙引號
